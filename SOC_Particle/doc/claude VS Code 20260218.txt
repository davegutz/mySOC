  1. mySum[NSUM] dominates volatile SRAM (LARGE)

  // constants.h:119
  #define NSUM 2400
  // SOC_Particle.ino:101
  Flt_st mySum[NSUM];

  Flt_st is 48 bytes (4 + 16×2 + 3×4 = 48). At NSUM=2400, this is 115,200 bytes — essentially the entire 119,852-byte data section. The compile    
  stats confirm this:

  text    data     bss     dec
  292998  119852   10306  423168

  The comments already warn that NSUM > ~2845 causes compile errors or SOS bus faults, meaning you're right at the edge. The 50-day rolling window 
  is useful, but every increment of NSUM costs 48 bytes of SRAM that can never be reclaimed.

  No obvious fix without changing the feature, but worth knowing this is the single biggest consumer.

  2. String objects in CommandPars — heap fragmentation risk (MODERATE)

  // command.h:55-61

  Seven String objects that grow and shrink with command input. On a Cortex-M33 heap this causes fragmentation. Command strings are short
  (typically under 32 chars). Fixed-size char[] buffers would eliminate this entirely and are equally readable.

  Changes to debug.h / debug.cpp — add_verify

  The fragmentation detector becomes a bounds checker:

  // Before — debug.h
  void add_verify(String *src, const String addend);

  // After — debug.h
  void add_verify(char *src, size_t src_size, const char *addend);

  // Before — debug.cpp
  void add_verify(String *src, const String addend)
  {
    int src_len = src->length();
    *src += addend;
    if ( src->length() != (src_len + addend.length()) )
      Serial.printf("\n\n\n\n**FRAG**\n\n\n\n");
  }

  // After — debug.cpp
  void add_verify(char *src, size_t src_size, const char *addend)
  {
    size_t src_len = strlen(src);
    size_t add_len = strlen(addend);
    if ( src_len + add_len + 1 > src_size )
      Serial.printf("\n\n\n\n**OVERFLOW**\n\n\n\n");
    else
      strncat(src, addend, src_size - src_len - 1);
  }

  Changes to serial.h / serial.cpp — chat_cmd_from

  The function currently returns a String built by consuming chars from the front of *source. It needs an output buffer parameter instead:

  // Before — serial.h
  String chat_cmd_from(String *source);

  // After — serial.h
  void chat_cmd_from(char *source, char *out, size_t out_size);

  // Before — serial.cpp
  String chat_cmd_from(String *source)
  {
    String out_str = "";
    while ( source->length() )
    {
      char in_char = source->charAt(0);
      source->remove(0, 1);
      out_str += in_char;
      if ( is_finished(in_char) ) {
        out_str = finish_request(out_str);
        break;
      }
    }
    return out_str;
  }

  // After — serial.cpp
  void chat_cmd_from(char *source, char *out, size_t out_size)
  {
    size_t out_idx = 0;
    out[0] = '\0';
    while ( *source )
    {
      char in_char = source[0];
      memmove(source, source + 1, strlen(source)); // shift left, includes '\0'
      if ( out_idx + 1 < out_size )
        out[out_idx++] = in_char;
      out[out_idx] = '\0';
      if ( is_finished(in_char) ) {
        finish_request(out, out_size);              // see note below
        break;
      }
    }
  }

  Note on finish_request: If it currently takes/returns a String, its signature must also change to void finish_request(char *s, size_t size) operating in-place.

  5. double arithmetic on a single-precision FPU (MODERATE — throughput/code size)

  The Photon2 (Cortex-M33) has a single-precision FPU only. All double operations are software-emulated (~10× slower, larger code). The codebase   
  mixes double and float throughout. Filter parameters, Kalman state variables, and timing calculations frequently use double. Literal constants   
  like 0.5, 0.80, 22.0 without f suffix are double by default. The impact on the 100ms sensor loop is real.

  12. Inconsistent spacing in conditionals (MINOR)

  13. unsigned long long vs uint64_t (MINOR)

  unsigned long long is used throughout for timing (millis_flip, last_sync, elapsed, now, etc.). uint64_t from <stdint.h> is more portable, more   
  self-documenting, and what the rest of the Particle embedded world uses.

  14. double(Sen->now/1000) integer-division bug risk (SUBTLE)

  Sen->now is unsigned long long. The division /1000 is integer division done before the cast to double, losing sub-second precision. If
  millisecond accuracy matters here, it should be double(Sen->now)/1000.0.
